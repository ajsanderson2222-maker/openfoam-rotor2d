# 2D Rotor вҖ” Dynamic Moving Mesh Simulation

A transient, turbulent simulation of a 4-bladed 2D rotor in a 1 m/s freestream using OpenFOAM 13. The rotor operates at 60 RPM with the entire computational domain rotating via a solid-body dynamic mesh (`motionSolver solidBody`). The case demonstrates dynamic mesh motion, rotating-frame boundary conditions, per-step torque and thrust extraction, and vortex wake visualisation.

## 1. Physical Problem

A 4-bladed rotor (propeller / fan cross-section) operates in a uniform freestream. Viewed from the front the rotor appears as four symmetric aerofoil sections arranged radially. The freestream flows in the +x direction at UвҲһ = 1 m/s while the rotor spins at 60 RPM about the z-axis (out of the page).

Key non-dimensional parameters:

| Parameter | Value | Formula |
|---|---|---|
| Freestream velocity UвҲһ | 1.0 m/s | вҖ” |
| Rotor speed | 60 RPM | вҖ” |
| Angular velocity Пү | 6.283 rad/s | 2ПҖ Г— 60/60 |
| Tip speed V_tip | 0.126 m/s | Пү Г— r_tip |
| Tip speed ratio О» | 0.126 | V_tip / UвҲһ |
| Rotor diameter D | 40 mm | 2 Г— r_tip |
| Reynolds number Re | 4 000 | UвҲһ Г— D / ОҪ |
| Rotational period T | 1.0 s | 60/RPM |
| Simulation duration | 2.0 s | 2 full rotations |
| Kinematic viscosity ОҪ | 1 Г— 10вҒ»вҒө mВІ/s | air at ~20 В°C |

The **tip speed ratio О» = 0.126** places this rotor in a lightly loaded, low-advance-ratio regime. At this О» the blade sections operate at a small effective angle of attack, generating modest thrust in the +x direction and a small resisting torque. A higher О» (achieved by increasing RPM or decreasing UвҲһ) would produce stronger blade loading and a more complex vortical wake.

## 4. Boundary Conditions

### Velocity (U)

| Patch | Type | Rationale |
|---|---|---|
| `rotor` (blade surfaces + hub) | `movingWallVelocity` | Applies u = u_mesh at the wall, where u_mesh is the instantaneous velocity of the rotating mesh face. This ensures the no-slip condition is satisfied in the inertial (lab) frame even though the mesh is moving. |
| `freestream` (outer circle) | `freestreamVelocity` | Applies the far-field condition: `zeroGradient` when flow exits, fixed `(1, 0, 0)` when flow enters. Correct for an open far-field boundary around a body in freestream. |
| `frontAndBack` | `empty` | 2D assumption |

**Why `movingWallVelocity` and not `noSlip`?**  
`noSlip` sets U = 0 in the frame of the mesh. For a stationary mesh that is correct. But when the mesh is rotating, the mesh face itself is moving at velocity u_mesh = Пү Г— r. The `movingWallVelocity` condition reads the local mesh point velocity and imposes u_fluid = u_mesh, correctly enforcing no-slip in the lab frame. Using `noSlip` on a moving mesh would set the fluid velocity to zero in the mesh frame вҖ” equivalent to a fixed wall in the rotating frame, which is incorrect for a dynamic mesh approach.

### Pressure (p, kinematic: mВІ/sВІ)

| Patch | Type |
|---|---|
| `rotor` | `zeroGradient` |
| `freestream` | `freestreamPressure` (p = 0 far field) |

### Turbulence (k, Оө, nut)

| Patch | k | Оө |
|---|---|---|
| `rotor` | `kqRWallFunction` | `epsilonWallFunction` |
| `freestream` | `inletOutlet` (k = 3.75 Г— 10вҒ»Ві mВІ/sВІ) | `inletOutlet` (Оө = 3.75 Г— 10вҒ»вҒҙ) |

Free-stream turbulence intensity: Tu = вҲҡ(2k/3) / UвҲһ вүҲ 5%, representative of a wind tunnel with mild upstream disturbances.

## 5. Dynamic Mesh вҖ” How Solid-Body Rotation Works

```
// constant/dynamicMeshDict
mover
{
    type                    motionSolver;
    motionSolver            solidBody;
    cellZone                all;
    solidBodyMotionFunction rotatingMotion;
    origin                  (0 0 0);
    axis                    (0 0 1);
    omega                   60 [rpm];
}
```

At every time step, OpenFOAM:

1. **Rotates all mesh points** by О”Оё = Пү Г— О”t = 6.283 Г— 2Г—10вҒ»вҒҙ = 0.00126 rad = **0.072В°** about the z-axis.
2. **Recomputes face areas and cell volumes** from the new point positions.
3. **Solves the flow equations** on the updated mesh, using the Arbitrary Lagrangian-Eulerian (ALE) form of the Navier-Stokes equations, which adds a mesh-velocity flux correction:

```
вҲӮ(ПҒU)/вҲӮt + вҲҮВ·(ПҒ(U вҲ’ u_mesh)U) = вҲ’вҲҮp + вҲҮВ·П„
```

where u_mesh is the local mesh point velocity. This correction ensures the Geometric Conservation Law (GCL) is satisfied вҖ” no spurious mass is created or destroyed by the moving mesh.

4. **Updates boundary condition values** вҖ” `movingWallVelocity` reads the new mesh face velocity from the motion solver and applies it as the wall BC.

The advantage of solid-body rotation over MRF is that **the flow field is computed in the inertial (lab) frame at every instant**. There is no need to add Coriolis or centrifugal source terms, and the solution is inherently time-accurate. The disadvantage is cost: the mesh must be rebuilt every step, adding overhead compared to MRF which solves once at steady state.

### Geometric Conservation Law

Moving meshes introduce a subtle numerical challenge: if the mesh moves but the discrete divergence theorem is not satisfied on the new geometry, spurious mass sources appear. OpenFOAM's `ddtSchemes Euler` formulation automatically satisfies the GCL by using the swept-volume flux correction at each face.

## 6. Comparison with MRF

This case complements the earlier **Rushton MRF** case in this portfolio, which used the Moving Reference Frame approach. The key differences:

| Feature | MRF (Rushton) | Dynamic Mesh (this case) |
|---|---|---|
| Mesh motion | None вҖ” mesh is fixed | Physical mesh rotation each step |
| Frame | Rotating reference frame | Inertial (lab) frame |
| Coriolis/centrifugal terms | Added as source terms | Not needed (naturally captured) |
| Steady/transient | Steady-state | Transient (time-accurate) |
| Rotor-stator interaction | Approximated (frozen rotor) | Exact (at each instant) |
| Computational cost | Low вҖ” one steady solve | High вҖ” full transient |
| Use case | Mean performance, impeller design | Blade passing, unsteady loading, wake |

MRF is appropriate when you want fast, steady-state performance metrics (power number, pressure rise). Dynamic mesh is needed when you want the **time-resolved blade loading**, unsteady pressure fluctuations on nearby structures, or an accurate vortex wake that accounts for blade passage effects.

## 7. Turbulence Model

```
// constant/momentumTransport
simulationType  RAS;
RAS { model kEpsilon; turbulence on; }
```

**k-Оө** (Jones & Launder 1972) is used for the turbulent fluid. At Re = 4 000 based on the rotor diameter, the flow in the blade passages and near wake is in the transitional-to-turbulent range. k-Оө is appropriate here because:

- The far-field and wake are free-shear-dominated regions where k-Оө performs well.
- The low tip speed (V_tip = 0.126 m/s) means blade boundary layers are thin and the wall functions (`kqRWallFunction`, `epsilonWallFunction`) adequately represent near-wall physics without resolving to yвҒә = 1.
- It is computationally cheaper than k-Пү SST for this domain size, and the bluff-body separation concerns that motivate SST are less critical for low-loaded flat-plate blades at low angle of attack.

## 8. Numerical Schemes and Solver

### Schemes

| Term | Scheme | Rationale |
|---|---|---|
| `ddt` | Euler (1st order implicit) | Unconditionally stable; adequate for time-accurate rotor loading (dominant frequency ~1 Hz) |
| `div(phi,U)` | `Gauss limitedLinearV 1` | Vector-limited linear; 2nd order with flux limiting near blade surfaces to prevent oscillations |
| `div(phi,k/Оө)` | `Gauss limitedLinear 1` | Bounded for turbulence scalars |
| `laplacian` | `Gauss linear corrected` | 2nd order with non-orthogonality correction |

### PIMPLE solver

```
PIMPLE
{
    nOuterCorrectors  1;    // single outer iteration вҖ” PISO mode
    nCorrectors       2;    // two PISO pressure correctors per step
    nNonOrthogonalCorrectors 0;  // mesh is orthogonal
}
```

With `nOuterCorrectors 1` the PIMPLE loop degenerates to standard **PISO**, which is correct for transient simulations where CFL < 1. PISO does not need outer iterations because each time step is small enough that the nonlinear coupling between U and p is weak.

### Linear solvers

| Field | Solver |
|---|---|
| `p` | GAMG + GaussSeidel smoother (symmetric, fast on structured mesh) |
| `U`, `k`, `Оө` | smoothSolver + GaussSeidel |

## 9. Time-Stepping and Run Control

```
deltaT    2e-4 s
endTime   2.0 s     вҶ’  2 full rotor rotations (T = 1.0 s/rev)
```

Each time step advances the rotor by **0.072В°** вҖ” 5 000 steps per revolution. This fine angular resolution ensures:

- The blade passage pressure pulse is well-resolved in time (1 blade passage = 90В°/rev = 1 250 steps).
- CFL < 0.2 throughout the domain (max CFL вүҲ 0.18 near blade tips).
- The ALE flux correction is applied accurately with small angular increments.

Two full rotations allow the transient startup (t вүҲ 0 вҖ“ 0.5 s) to decay and the periodic wake shedding to establish. Statistics and animations are taken from the second rotation (t = 1 вҖ“ 2 s).

**Wall-clock time:** ~5 minutes on a single core for 10 000 time steps.

## 11. Results

### Force and moment time histories

The torque M_z(t) and thrust F_x(t) oscillate at the **blade passing frequency** f_bp = nBlades Г— RPM/60 = 4 Г— 1 = 4 Hz (one pulse per blade passing the reference angle). After startup transients decay (~0.5 s), both signals settle into a periodic steady state synchronised with the rotation.

### Wake velocity probes

The three wake probes (1D, 2D, 3D downstream) show the **velocity deficit** behind the rotor. As the rotor passes, each probe records a periodic dip in Ux corresponding to the low-momentum wake shed by each blade. The deficit recovers with downstream distance as turbulent mixing re-energises the wake.

### Vorticity animation

The vorticity field (Пүz component) shows **tip vortices** shed from each blade tip as it rotates. These form a helicoidal pattern in the wake, characteristic of low-tip-speed-ratio rotors. The counter-rotating vortices from adjacent blades interact and pair in the near wake.

## 13. How to Run

- OpenFOAM 13 (sourced)
- Python 3 with `pyvista`, `matplotlib`, `numpy`, `scipy`, `imageio`

```bash

pip install pyvista matplotlib numpy scipy imageio
```

```bash

```

Runs to t = 2 s (2 full rotations) in ~5 minutes on a single core.
