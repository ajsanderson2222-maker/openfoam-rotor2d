# 2D Rotor — Dynamic Moving Mesh Simulation

A transient, turbulent simulation of a 4-bladed 2D rotor in a 1 m/s freestream using OpenFOAM 13. The rotor operates at 60 RPM with the entire computational domain rotating via a solid-body dynamic mesh (`motionSolver solidBody`). The case demonstrates dynamic mesh motion, rotating-frame boundary conditions, per-step torque and thrust extraction, and vortex wake visualisation.

## 1. Physical Problem

A 4-bladed rotor (propeller / fan cross-section) operates in a uniform freestream. Viewed from the front the rotor appears as four symmetric aerofoil sections arranged radially. The freestream flows in the +x direction at U∞ = 1 m/s while the rotor spins at 60 RPM about the z-axis (out of the page).

Key non-dimensional parameters:

| Parameter | Value | Formula |
|---|---|---|
| Freestream velocity U∞ | 1.0 m/s | — |
| Rotor speed | 60 RPM | — |
| Angular velocity ω | 6.283 rad/s | 2π × 60/60 |
| Tip speed V_tip | 0.126 m/s | ω × r_tip |
| Tip speed ratio λ | 0.126 | V_tip / U∞ |
| Rotor diameter D | 40 mm | 2 × r_tip |
| Reynolds number Re | 4 000 | U∞ × D / ν |
| Rotational period T | 1.0 s | 60/RPM |
| Simulation duration | 2.0 s | 2 full rotations |
| Kinematic viscosity ν | 1 × 10⁻⁵ m²/s | air at ~20 °C |

The **tip speed ratio λ = 0.126** places this rotor in a lightly loaded, low-advance-ratio regime. At this λ the blade sections operate at a small effective angle of attack, generating modest thrust in the +x direction and a small resisting torque. A higher λ (achieved by increasing RPM or decreasing U∞) would produce stronger blade loading and a more complex vortical wake.

## 2. Domain and Geometry

The domain is a circular disk of radius r_freestream = 100 mm extruded to a thin 2D slab (thickness = 2 × halfDepth = 10 mm). The front and back faces are `empty` patches for the 2D assumption.

```
          r_freestream = 100 mm
    ┌────────────────────────────┐
    │      freestream BC         │
    │    ╔══════════════╗        │
    │    ║  r_tip=20mm  ║        │
    │    ║    /blade\   ║  → U∞  │
    │    ║  ○  centre   ║        │
    │    ║    \blade/   ║        │
    │    ╚══════════════╝        │
    │       r_hub=5mm            │
    └────────────────────────────┘
```

The geometry is parameterised in `system/blockMeshDict`:

| Parameter | Value |
|---|---|
| `nBlades` | 4 |
| `rHub` | 5 mm |
| `rTip` | 20 mm |
| `rFreestream` | 100 mm |
| `halfDepth` | 5 mm |

The blade sections are flat plates generated by the `codeStream` block mesh, arranged symmetrically around the hub. Four identical blade passages are created between `rHub` and `rTip`, each separated by `360°/nBlades = 90°`.

## 4. Boundary Conditions

### Velocity (U)

| Patch | Type | Rationale |
|---|---|---|
| `rotor` (blade surfaces + hub) | `movingWallVelocity` | Applies u = u_mesh at the wall, where u_mesh is the instantaneous velocity of the rotating mesh face. This ensures the no-slip condition is satisfied in the inertial (lab) frame even though the mesh is moving. |
| `freestream` (outer circle) | `freestreamVelocity` | Applies the far-field condition: `zeroGradient` when flow exits, fixed `(1, 0, 0)` when flow enters. Correct for an open far-field boundary around a body in freestream. |
| `frontAndBack` | `empty` | 2D assumption |

**Why `movingWallVelocity` and not `noSlip`?**  
`noSlip` sets U = 0 in the frame of the mesh. For a stationary mesh that is correct. But when the mesh is rotating, the mesh face itself is moving at velocity u_mesh = ω × r. The `movingWallVelocity` condition reads the local mesh point velocity and imposes u_fluid = u_mesh, correctly enforcing no-slip in the lab frame. Using `noSlip` on a moving mesh would set the fluid velocity to zero in the mesh frame — equivalent to a fixed wall in the rotating frame, which is incorrect for a dynamic mesh approach.

### Pressure (p, kinematic: m²/s²)

| Patch | Type |
|---|---|
| `rotor` | `zeroGradient` |
| `freestream` | `freestreamPressure` (p = 0 far field) |

### Turbulence (k, ε, nut)

| Patch | k | ε |
|---|---|---|
| `rotor` | `kqRWallFunction` | `epsilonWallFunction` |
| `freestream` | `inletOutlet` (k = 3.75 × 10⁻³ m²/s²) | `inletOutlet` (ε = 3.75 × 10⁻⁴) |

Free-stream turbulence intensity: Tu = √(2k/3) / U∞ ≈ 5%, representative of a wind tunnel with mild upstream disturbances.

## 5. Dynamic Mesh — How Solid-Body Rotation Works

```
// constant/dynamicMeshDict
mover
{
    type                    motionSolver;
    motionSolver            solidBody;
    cellZone                all;
    solidBodyMotionFunction rotatingMotion;
    origin                  (0 0 0);
    axis                    (0 0 1);
    omega                   60 [rpm];
}
```

At every time step, OpenFOAM:

1. **Rotates all mesh points** by Δθ = ω × Δt = 6.283 × 2×10⁻⁴ = 0.00126 rad = **0.072°** about the z-axis.
2. **Recomputes face areas and cell volumes** from the new point positions.
3. **Solves the flow equations** on the updated mesh, using the Arbitrary Lagrangian-Eulerian (ALE) form of the Navier-Stokes equations, which adds a mesh-velocity flux correction:

```
∂(ρU)/∂t + ∇·(ρ(U − u_mesh)U) = −∇p + ∇·τ
```

where u_mesh is the local mesh point velocity. This correction ensures the Geometric Conservation Law (GCL) is satisfied — no spurious mass is created or destroyed by the moving mesh.

4. **Updates boundary condition values** — `movingWallVelocity` reads the new mesh face velocity from the motion solver and applies it as the wall BC.

The advantage of solid-body rotation over MRF is that **the flow field is computed in the inertial (lab) frame at every instant**. There is no need to add Coriolis or centrifugal source terms, and the solution is inherently time-accurate. The disadvantage is cost: the mesh must be rebuilt every step, adding overhead compared to MRF which solves once at steady state.

### Geometric Conservation Law

Moving meshes introduce a subtle numerical challenge: if the mesh moves but the discrete divergence theorem is not satisfied on the new geometry, spurious mass sources appear. OpenFOAM's `ddtSchemes Euler` formulation automatically satisfies the GCL by using the swept-volume flux correction at each face.

## 6. Comparison with MRF

This case complements the earlier **Rushton MRF** case in this portfolio, which used the Moving Reference Frame approach. The key differences:

| Feature | MRF (Rushton) | Dynamic Mesh (this case) |
|---|---|---|
| Mesh motion | None — mesh is fixed | Physical mesh rotation each step |
| Frame | Rotating reference frame | Inertial (lab) frame |
| Coriolis/centrifugal terms | Added as source terms | Not needed (naturally captured) |
| Steady/transient | Steady-state | Transient (time-accurate) |
| Rotor-stator interaction | Approximated (frozen rotor) | Exact (at each instant) |
| Computational cost | Low — one steady solve | High — full transient |
| Use case | Mean performance, impeller design | Blade passing, unsteady loading, wake |

MRF is appropriate when you want fast, steady-state performance metrics (power number, pressure rise). Dynamic mesh is needed when you want the **time-resolved blade loading**, unsteady pressure fluctuations on nearby structures, or an accurate vortex wake that accounts for blade passage effects.

## 7. Turbulence Model

```
// constant/momentumTransport
simulationType  RAS;
RAS { model kEpsilon; turbulence on; }
```

**k-ε** (Jones & Launder 1972) is used for the turbulent fluid. At Re = 4 000 based on the rotor diameter, the flow in the blade passages and near wake is in the transitional-to-turbulent range. k-ε is appropriate here because:

- The far-field and wake are free-shear-dominated regions where k-ε performs well.
- The low tip speed (V_tip = 0.126 m/s) means blade boundary layers are thin and the wall functions (`kqRWallFunction`, `epsilonWallFunction`) adequately represent near-wall physics without resolving to y⁺ = 1.
- It is computationally cheaper than k-ω SST for this domain size, and the bluff-body separation concerns that motivate SST are less critical for low-loaded flat-plate blades at low angle of attack.

## 8. Numerical Schemes and Solver

### Schemes

| Term | Scheme | Rationale |
|---|---|---|
| `ddt` | Euler (1st order implicit) | Unconditionally stable; adequate for time-accurate rotor loading (dominant frequency ~1 Hz) |
| `div(phi,U)` | `Gauss limitedLinearV 1` | Vector-limited linear; 2nd order with flux limiting near blade surfaces to prevent oscillations |
| `div(phi,k/ε)` | `Gauss limitedLinear 1` | Bounded for turbulence scalars |
| `laplacian` | `Gauss linear corrected` | 2nd order with non-orthogonality correction |

### PIMPLE solver

```
PIMPLE
{
    nOuterCorrectors  1;    // single outer iteration — PISO mode
    nCorrectors       2;    // two PISO pressure correctors per step
    nNonOrthogonalCorrectors 0;  // mesh is orthogonal
}
```

With `nOuterCorrectors 1` the PIMPLE loop degenerates to standard **PISO**, which is correct for transient simulations where CFL < 1. PISO does not need outer iterations because each time step is small enough that the nonlinear coupling between U and p is weak.

### Linear solvers

| Field | Solver |
|---|---|
| `p` | GAMG + GaussSeidel smoother (symmetric, fast on structured mesh) |
| `U`, `k`, `ε` | smoothSolver + GaussSeidel |

## 9. Time-Stepping and Run Control

```
deltaT    2e-4 s
endTime   2.0 s     →  2 full rotor rotations (T = 1.0 s/rev)
```

Each time step advances the rotor by **0.072°** — 5 000 steps per revolution. This fine angular resolution ensures:

- The blade passage pressure pulse is well-resolved in time (1 blade passage = 90°/rev = 1 250 steps).
- CFL < 0.2 throughout the domain (max CFL ≈ 0.18 near blade tips).
- The ALE flux correction is applied accurately with small angular increments.

Two full rotations allow the transient startup (t ≈ 0 – 0.5 s) to decay and the periodic wake shedding to establish. Statistics and animations are taken from the second rotation (t = 1 – 2 s).

**Wall-clock time:** ~5 minutes on a single core for 10 000 time steps.

## 10. Post-processing Function Objects

Defined in `system/functions`:

| Object | Output |
|---|---|
| `rotorForces` | Pressure + viscous forces and moments on blade surfaces, written every step → thrust F_x(t), torque M_z(t) |
| `rotorForceCoeffs` | Non-dimensional Ct (thrust) and Cm (torque) coefficients referenced to U∞, D, A_ref |
| `wakeProbes` | U and p at x = 1D, 2D, 3D downstream on the rotor axis — velocity deficit recovery |
| `vorticity` | ω = ∇×U field written every 5 steps for wake animation |
| `residuals` | p and U residuals for convergence monitoring |

Reference values for force coefficients:

```
ρ_ref  = 1.225 kg/m³  (air)
U_ref  = 1.0 m/s      (freestream)
l_ref  = D = 40 mm    (rotor diameter)
A_ref  = 2·r_tip × 2·halfDepth = 4×10⁻⁴ m²
```

## 11. Results

### Performance summary

![KPI table](kpi_table.png)

### Force and moment time histories

![Force and torque time series](forces.png)

The torque M_z(t) and thrust F_x(t) oscillate at the **blade passing frequency** f_bp = nBlades × RPM/60 = 4 × 1 = 4 Hz (one pulse per blade passing the reference angle). After startup transients decay (~0.5 s), both signals settle into a periodic steady state synchronised with the rotation. The negative mean torque (Cm = −0.035) confirms the rotor is aerodynamically resisted — the motor must do work to maintain rotation against the fluid drag, as expected at this low TSR.

### Wake velocity deficit

![Wake velocity probes](wake_velocity.png)

The two in-domain wake probes (1D and 2D downstream) show the **velocity deficit** behind the rotor. As the rotor passes, each probe records a periodic dip in Ux corresponding to the low-momentum wake shed by each blade. The deficit recovers with downstream distance as turbulent mixing re-energises the wake.

### Vorticity field (t = 2 s)

![Vorticity snapshot at t = 2 s](vorticity_snapshot.png)

The vorticity field (ωz component) shows **tip vortices** shed from each blade tip as it rotates. These form a helicoidal pattern in the wake, characteristic of low-tip-speed-ratio rotors. The counter-rotating vortices from adjacent blades interact and pair in the near wake. Red = positive (counter-clockwise) vorticity; blue = negative (clockwise).

## 12. Future Improvements

| Item | Impact | Effort |
|---|---|---|
| **Increase RPM / TSR** — raise to 300–600 RPM (λ ≈ 0.6–1.3) | Stronger blade loading, larger wake momentum deficit, more vivid tip vortices | Low (change omega in dynamicMeshDict) |
| **Proper aerofoil blade profile** — replace flat-plate geometry with NACA 0012 cross-sections via snappyHexMesh | Realistic lift/drag polar, attached boundary layer, stall behaviour | High |
| **AMI sliding interface** — separate inner rotor zone and outer stator zone with an Arbitrary Mesh Interface | True rotor-stator interaction; allows fixed outer domain (e.g. duct walls, guide vanes) without rotating the whole mesh | High |
| **3D rotor** — extend blade geometry into a full axial fan/propeller with spanwise twist and chord variation | 3D tip vortex structure, spanwise loading distribution, Prandtl tip-loss correction comparison | Very high |
| **Blade Element Momentum (BEM) comparison** — post-process blade loading and compare Ct vs λ to BEM theory | Quantitative validation of thrust coefficient across tip speed ratios | Medium |
| **Finer mesh near blade** — reduce first cell height to y⁺ ≈ 1 and use low-Re k-ω SST instead of wall functions | Resolve laminar sublayer; more accurate blade surface pressure and friction drag | Medium |
| **Extend to 5–10 rotations** — run longer to gather phase-averaged statistics | Reduce statistical noise in force/moment spectra; compute phase-averaged wake profiles | Low (increase endTime) |

## 13. How to Run

### Prerequisites

- OpenFOAM 13 (sourced)
- Python 3 with `pyvista`, `matplotlib`, `numpy`, `scipy`, `imageio`

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install pyvista matplotlib numpy scipy imageio
```

### Mesh

```bash
blockMesh    # structured parametric multi-block mesh
checkMesh    # verify quality (should report Mesh OK)
```

### Run

```bash
foamRun      # transient incompressibleFluid + solidBody dynamic mesh
```

Runs to t = 2 s (2 full rotations) in ~5 minutes on a single core.

### Post-process

```bash
foamToVTK
source .venv/bin/activate
python postprocess.py
```
